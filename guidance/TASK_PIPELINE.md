# 交办流程

全局默认开发流程。所有 `/交办` 任务按此流程执行，阶段间设确认门禁，未经用户确认不得进入下一阶段。

## 流程总览

```
交办启动
  │
  ▼
┌─────────────────┐
│ 阶段1：需求澄清   │
│ 产出：需求确认单  │     .claude/tasks/<task>/需求确认单.md
└────────┬────────┘
         │ ⛔ 用户确认需求
         ▼
┌─────────────────┐
│ 阶段2：技术方案   │
│ 产出：方案+任务单 │     .claude/tasks/<task>/技术方案.md
│       +接口协议   │     .claude/tasks/<task>/接口协议.md
└────────┬────────┘
         │ ⛔ 用户确认方案
         ▼
┌─────────────────┐
│ 阶段3：Spec 撰写  │
│ 产出：变更提案    │     openspec/ 下的提案+设计+规格+任务清单
│ 缺少 OpenSpec？   │     → 询问用户是否自动安装
└────────┬────────┘
         │ ⛔ 用户确认 Spec
         ▼
┌─────────────────┐
│ 阶段4：代码实现   │
│ 产出：代码变更    │     按任务清单逐条实现
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 阶段5：代码审查   │
│ 修复→再审→通过   │     多轮循环
└────────┬────────┘
         │ ⛔ 用户确认审查通过
         ▼
┌─────────────────┐
│ 阶段6：交付通知   │
│ 变更摘要+审查结论 │     .claude/tasks/<task>/交付通知.md
└─────────────────┘
```

## 阶段说明

### 阶段1：需求澄清
- 目的：对齐需求理解，澄清模糊点
- 产出：`需求确认单.md`，包含功能点、边界条件、与现有系统的关系
- 门禁：用户明确确认需求无误后，进入阶段2

### 阶段2：技术方案
- 目的：设计实现方案，拆解任务
- 产出：
  - `技术方案.md`：数据库变更、接口设计、文件清单、编码顺序
  - `接口协议.md`：给前端的完整接口协议，含路径、请求体字段、响应格式、示例
- 门禁：用户明确确认方案无误后，进入阶段3

### 阶段3：Spec 撰写
- 目的：将技术方案转化为 OpenSpec 变更提案，作为代码实现的规格约束，防止遗漏和偏差
- 前置检查：OpenSpec 是否可用，不可用则询问用户是否自动安装
- 产出：`openspec/` 下的变更提案（提案 + 设计 + 规格 + 任务清单）
- 门禁：用户明确确认 Spec 无问题后，进入阶段4

### 阶段4：代码实现
- 目的：按 Spec 任务清单逐条编码
- 产出：代码变更（commit）
- 完成后自动进入阶段5

### 阶段5：代码审查
- 目的：发现功能、编译、安全、风格问题
- 方式：审查 → 发现问题 → 修复 → 再审，多轮循环
- 门禁：用户明确确认审查通过后，进入阶段6

### 阶段6：交付通知
- 目的：汇总变更内容和审查结论，通知用户

## 任务文档目录

每个任务在 `.claude/tasks/<task-name>/` 下创建独立目录，所有阶段文档存放于此。这是 WHAT（工作产物），跟随项目。

```
.claude/tasks/<task-name>/
├── 流程执行状态.json            # 流程执行状态，实时更新
├── 需求确认单.md
├── 技术方案.md
├── 接口协议.md
├── 交付通知.md
└── ...
```

## 状态追踪

每个任务维护 `流程执行状态.json`，记录执行进度：

```json
{
  "task_name": "<task-name>",
  "status": "in_progress",
  "current_stage": 3,
  "stages": {
    "1": {"name": "需求澄清", "status": "completed", ...},
    "2": {"name": "技术方案", "status": "completed", ...},
    "3": {"name": "Spec撰写", "status": "in_progress", ...},
    "4": {"name": "代码实现", "status": "pending"},
    "5": {"name": "代码审查", "status": "pending"},
    "6": {"name": "交付通知", "status": "pending"}
  },
  "created_at": "ISO时间",
  "updated_at": "ISO时间"
}
```

- 进入阶段 → status 改为 `in_progress`，记录 `started_at`
- 通过门禁 → status 改为 `completed`，记录 `completed_at`
- 6 阶段全部完成 → 任务 status 改为 `completed`

## 门禁规则

- 每个 ⛔ 标记的确认点，必须等待用户明确说"通过"/"确认"/"没问题"等，才能进入下一阶段
- 用户未确认时，不得擅自推进
- 若用户中途修改需求，回到阶段1重新对齐

## 优先级

1. `.claude/guidance/TASK_PIPELINE.md`（项目级覆盖）
2. `~/.claude/agent-manager/TASK_PIPELINE.md`（用户级）
3. 本文件（插件默认）

对话中修改流程时，manager 会询问"当前项目 or 全部项目"，分别写入对应位置。
