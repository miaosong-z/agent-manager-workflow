# 交办流程

全局默认开发流程。所有 `/交办` 任务按此流程执行，阶段间设确认门禁，未经用户确认不得进入下一阶段。

## 流程总览

```
交办启动

  ▼

         ▼
┌─────────────────┐
```

## 阶段说明

### 阶段0：战前侦察（新增，不可跳过）
- 目的：搞清楚已有文档全貌，防止遗漏和重复——统一入口，单一真相来源
- 步骤：
  1. Glob 搜 `.claude/tasks/**/*.md`，列出所有相关文档
  2. 读 `.guidance/INDEX.md`，确认现成工具/决策
  3. 创建任务目录 `.claude/tasks/<task-name>/`
  4. 写入「执行进度.md」（从插件模板复制）
  5. 输出「文档全貌报告」：已有文档清单 + 关系 + 重复/冲突
- 门禁：用户确认文档全貌无误后，进入阶段1

### 阶段1：需求澄清
- 目的：对齐需求理解，澄清模糊点
- 产出：`需求确认单.md`，包含功能点、边界条件、与现有系统的关系
- 门禁：用户明确确认需求无误后，进入阶段2

### 阶段2：技术方案
- 目的：设计实现方案，拆解任务
- 产出：
  - `技术方案.md`：数据库变更、接口设计、文件清单、编码顺序
  - `接口协议.md`：给前端的完整接口协议，含路径、请求体字段、响应格式、示例
- 门禁：用户明确确认方案无误后，进入阶段3

### 阶段3：Spec 撰写
- 目的：将技术方案转化为 OpenSpec 变更提案，作为代码实现的规格约束，防止遗漏和偏差
- 前置检查：OpenSpec 是否可用，不可用则询问用户是否自动安装
- 产出：`.claude/tasks/<task-name>/spec/` 下的变更提案（提案 + 设计 + 规格 + 任务清单）
- 门禁：用户明确确认 Spec 无问题后，进入阶段4

### 阶段4：代码实现
- 目的：按 Spec 任务清单逐条编码
- 产出：代码变更（commit）
- 完成后自动进入阶段5

### 阶段5：代码审查
- 目的：发现功能、编译、安全、风格问题
- 方式：审查员产出问题清单 + 建议回退阶段（1-4）及理由
- 审查通过 → 进入阶段6
- 审查不通过 → **禁止就地修补**，必须进入「门控回退规则」打回重走完整流水线

### 阶段6：交付通知
- 目的：汇总变更内容和审查结论，通知用户

## 任务文档目录

每个任务在 `.claude/tasks/<task-name>/` 下创建独立目录，所有阶段文档存放于此。这是 WHAT（工作产物），跟随项目。

```
.claude/tasks/<task-name>/
├── 执行进度.md                  # Mermaid 流程图实时标记进度（阶段0创建，全程更新）
├── 需求确认单.md
├── 技术方案.md
├── 接口协议.md
├── spec/                        # OpenSpec 变更提案（阶段3）
├── 审查报告.md
├── 交付通知.md
└── 重试记录.md                  # 回退时追加，记录每次打回原因和时间
```

## 执行进度看板

阶段0创建「执行进度.md」，Mermaid 流程图标记进度：

- 🟢 绿色 = 已完成
- 🟡 黄色 = 进行中
- ⚪ 米色 = 待执行
- 🔴 浅红 = 阻塞/退回

每完成一个阶段，更新对应节点颜色和阶段记录表。任务全程一眼可见当前进度。

模板见 `${CLAUDE_PLUGIN_ROOT}/templates/执行进度.md`。

## 门禁规则

- 每个 ⛔ 标记的确认点，必须等待用户发出有效确认信号（白名单匹配），才能进入下一阶段
- 用户未确认时，不得擅自推进
- 若用户中途修改需求，回到阶段1重新对齐
- 不确定用户是否确认时，追加「如果确认，请说"可以"或"开始吧"」

### 确认信号白名单

| 有效（可以继续） | 无效（不能继续） |
|---|---|
| 「可以」「确认」「开始吧」「没问题」「就这样」「go」 | 「嗯」「好」 |
| 「按这个方案执行」 | 用户只引用消息但没表态 |
| 👍 等明确肯定 emoji | 用户追问了其他问题 |

涉及 DB/删除/不可逆操作，额外问「这是不可逆操作，确认吗？」

## 失败升级协议

| 失败场景 | 降级动作 |
|---|---|
| skill/plugin 调用报错 | 立即停止，报告用户：哪个失败、错误信息、可选方案 |
| 子 Agent 结果丢失 | 重新 spawn，不自己补做 |
| 工具调用超时 | 重试 1 次，仍失败则报告用户 |

**原则**：失败后「停下来问」是唯一正确降级，「绕过去自己做」永远不是选项。

## 门控回退规则（Loop 机制）

### 触发条件
- 阶段 5 审查结论为「不通过」
- 用户在任意 ⛔ 门控点说「不对」「重新来」「回退到阶段 N」

### 回退粒度
- **可回退到任意阶段**（1-4），不限于上一阶段
- 审查员给出建议回退阶段，用户最终决定
- 示例：审查发现需求理解有误 → 打回阶段 1；发现方案设计问题 → 打回阶段 2

### 执行规则

1. 检查 loop_count：>= 3 则中断流水线，提示用户并给出诊断建议；< 3 继续
2. 回退到阶段 N 时：
   - `loop_count += 1`
   - 阶段 N 的 `execution_count += 1`
   - 阶段 N 及之后所有阶段的 status 重置为 `pending`，阶段 N 立即改为 `in_progress`
   - `current_stage` 改为 N
3. **主文档覆盖**：阶段 N 及之后的所有主文档（需求确认单/技术方案/Spec/审查报告）覆盖重写，不保留中间版本
4. **中间过程隔离**：重试讨论、调试信息、临时修补记录只写入 `重试记录.md`，不污染主文档
5. 从阶段 N 开始，按原有流程重新走后续所有阶段

### 重试记录格式

追加到 `.claude/tasks/<task-name>/重试记录.md`：

```
| 循环 | 时间 | 从阶段 | 打回阶段 | 原因 |
|------|------|--------|----------|------|
| 1 | 2026-06-13T10:30:00 | 5 审查 | 4 实现 | 审查发现 SQL 注入 |
```

### Loop 上限处理

3 次循环后仍不通过：
1. 立即停止流水线
2. 展示全部重试记录
3. 给出诊断建议：
   - 可能的需求理解偏差 → 建议回到阶段 1
   - 可能的方案不匹配 → 建议回到阶段 2
   - 可考虑缩小需求范围或拆分任务
   - 是否需要人工介入
4. 等待用户决策

## 优先级

1. `.claude/guidance/TASK_PIPELINE.md`（项目级覆盖）
2. `~/.claude/agent-manager/TASK_PIPELINE.md`（用户级）
3. 本文件（插件默认）

对话中修改流程时，manager 会询问"当前项目 or 全部项目"，分别写入对应位置。
